# 부록 5 — AI 프롬프팅 예시

> 같은 질문을 현장의 눈과 관리의 눈에 동시에 던져서, 답이 갈리는 지점을 찾는 연습이다 — 그 지점이 빚의 소재지다.

> **사용법**: 각 장마다 🔲 **보편 템플릿**이 먼저 나온다. `[___]`에 자기 사업을 넣으면 바로 쓸 수 있다. 그 아래 📋 **풍국산업 예시**는 "이렇게 채우면 된다"의 완성본이다.
> **시연 기업**: [[풍국산업]] × [[PuzzleData]] 협업 (가방 제조 SCM)
> **도구**: Claude

---

## 배경: 풍국산업이란?

- 가방 OEM 제조업체. 바이어(Adidas 등)의 주문을 받아 생산·납품
- 어맨드(사양 변경) 연 30회 — 바이어가 확정 후에도 사양을 바꿈
- BOM/BOP 이중 장부 — 개발용(바이어 가격 협상)과 생산용(실제 공정)이 의도적으로 다름
- 용어 혼란 — 영어·한국어·일본어 혼용. "같은 공정을 다른 이름으로" 부름
- [[PuzzleData]]: 프로세스 마이닝 기업. 풍국의 데이터를 구조화하는 파트너

## 1장 🪞 거울새장 — 욕망 Dashboard

> **병**: 원하니까 된다고 믿음
> **약**: 장부를 펴라

### 🔲 보편 템플릿

**D — 현장의 눈:**
```
나는 [내 회사/프로젝트]의 [현장 역할]이야.
우리 팀이 추진하는 아이디어 3개가 있어:
1. [아이디어 1]  2. [아이디어 2]  3. [아이디어 3]
각 아이디어가 현장의 어떤 고통에서 나왔는지,
내가 직접 겪은 문제인지 표로 만들어줘.
```

**S — 관리의 눈:**
```
나는 [내 회사/프로젝트]의 [분석/기획 역할]이야.
같은 아이디어 3개의 출처를 추적해줘.
내부 데이터인지 외부 벤치마킹인지,
사라져도 괜찮은지 — 빚인가 자산인가 표로 만들어줘.
```

**대조:** D의 표와 S의 표를 나란히 놓는다 — 답이 다른 곳이 빚의 소재지.

### 📋 풍국산업 예시

### 프롬프트 — D (현장 베테랑의 눈)

```
나는 풍국산업의 생산부장이야. 현장 20년차.
우리 팀이 추진하는 개선 아이디어 3개가 있어:

1. BOM/BOP 이중 장부를 하나로 통합하자
2. 어맨드 횟수를 연 30회에서 15회로 줄이자
3. PuzzleData의 프로세스 마이닝으로 병목을 자동 탐지하자

각 아이디어에 대해:
- 이 아이디어가 현장의 어떤 고통에서 나왔는지 설명해줘.
- 내가 직접 겪은 문제에서 비롯된 건지, 남의 성공 사례를 따라하는 건지 구분해줘.
- 표로 만들어줘: 아이디어 / 현장 고통 / 이건 내 것인가
```

### Claude 답변 D

| 아이디어 | 현장 고통 | 내 것인가 |
|:--|:--|:--|
| BOM/BOP 통합 | 매일 장부 두 개를 대조하며 야근한다. 수기 오류로 바이어 클레임이 분기 2~3건 | **내 것** — 내 손이 두 번 움직이는 고통에서 나왔다 |
| 어맨드 15회 축소 | 어맨드 때마다 생산 라인이 멈추고 재편하는 데 하루가 날아간다 | **내 것** — 라인이 멈추는 공포를 매번 겪는다 |
| 프로세스 마이닝 병목 탐지 | 봉제실 앞에서 3일이 사라지는데 원인을 모른다 | **내 것** — 어디서 시간이 새는지 모르는 답답함 |

### 프롬프트 — S (경영기획실 분석가의 눈)

```
나는 풍국산업의 경영기획실 과장이야.
현장에서 올라온 개선 아이디어 3개의 출처를 추적하려고 해:

1. BOM/BOP 이중 장부를 하나로 통합하자
2. 어맨드 횟수를 연 30회에서 15회로 줄이자
3. PuzzleData의 프로세스 마이닝으로 병목을 자동 탐지하자

각 아이디어에 대해:
- 이 아이디어의 원래 근거가 뭐였는지 추적해줘. 내부 데이터인가, 외부 벤치마킹인가, 컨설팅 보고서인가.
- 이 아이디어가 내일 사라져도 풍국은 괜찮은가?
- 표로 만들어줘: 아이디어 / 원래 근거 / 사라져도 괜찮은가 / 빚인가 자산인가
```

### Claude 답변 S

| 아이디어 | 원래 근거 | 사라져도 괜찮은가 | 빚/자산 |
|:--|:--|:--|:--|
| BOM/BOP 통합 | **빌린 것** — "SSOT가 정답"이라는 외부 컨설팅 논리. 그러나 풍국의 이중 장부는 가격 협상력의 원천. 통합하면 경쟁력이 무너질 수 있다 | 사라져도 괜찮다 — 오히려 "매핑"이 답 | **빚** — 검증 안 된 가정("하나가 좋다") 위에 서 있음 |
| 어맨드 15회 축소 | **빌린 것** — "변경을 줄이면 효율적"이라는 린 제조 교과서. 그러나 어맨드 = 바이어 신뢰 = 매출. "줄여라"는 "매출을 줄여라"와 같다 | 사라지면 위험 — 진짜 문제(전파 속도)를 놓침 | **빚** — 진짜 문제는 횟수가 아니라 전파 지연 |
| 프로세스 마이닝 병목 탐지 | **내 것** — 현장에서 "어디서 3일이 사라지는지 모른다"를 직접 겪음 | 사라지면 아쉽다 — 진짜 내 문제 | **자산** — 현장 경험에서 나온 질문 |

### 출처 대조표 — 차이가 신호다

| 아이디어 | D(현장)가 본 것 | S(기획)가 본 것 | 같은가 다른가? |
|:--|:--|:--|:--|
| BOM/BOP 통합 | 야근의 고통 → 내 것 | SSOT 논리 → 빌린 것 | **다름 = 고통은 진짜, 해법은 빌렸다** |
| 어맨드 축소 | 라인 정지 공포 → 내 것 | 린 교과서 → 빌린 것 | **다름 = 고통은 진짜, 처방이 빌렸다** |
| 병목 탐지 | 답답함 → 내 것 | 현장 경험 → 내 것 | **같음 = 맑은 하늘. 여기서 시작** |

### 이 실습이 바꾸는 것

- **관리속도↑**: 3개 아이디어의 출처를 머릿속에서 맴돌리던 것 → 5분 만에 표로 정리
- **관리정확도↑**: D만 들으면 "셋 다 내 것"인데, S를 옆에 놓으면 "고통은 내 것이되 해법은 빌렸다"가 드러난다. 관점이 하나뿐이면 빌린 줄도 모른다 — 그게 엠마의 거울이다

## 2장 🔭 망원경새장 — D vs S 모의 논쟁

> **병**: 다 보이니 원할 이유가 없음
> **약**: 순서를 타라

### 🔲 보편 템플릿

**D — 현장의 눈:**
```
나는 [내 회사/프로젝트]의 [현장 역할]이야.
[도입하려는 변화]를 추진하려고 해.
너는 현장 경험으로 말하는 조언자야.
왜 지금 움직여야 하는지, 내 눈으로 본 것 기준으로 3가지를 짚어줘.
```

**S — 관리의 눈:**
```
나는 [내 회사/프로젝트]의 [분석/기획 역할]이야.
[같은 변화]를 도입하려고 해.
너는 데이터와 규정의 관점에서 말하는 조언자야.
현장이 말하는 것과 데이터가 다를 수 있는 이유 3가지,
검증 순서를 되돌릴 수 있는 것부터 알려줘.
```

**대조:** D와 S가 같은 말을 하는 곳 = 맑은 하늘. 다른 곳 = 안개. 안개가 먼저 시험할 곳.

### 📋 풍국산업 예시

### 프롬프트 — D (생산부장 — 현장 20년의 눈)

```
나는 풍국산업의 생산부장이야. 현장 20년차.
PuzzleData와 협업해서 프로세스 마이닝을 도입하려고 해.
너는 현장 경험으로 말하는 조언자야.

이 도입이 왜 필요한지, 내가 매일 겪는 현장의 관점에서 3가지를 짚어줘.
- 내 눈으로 본 병목이 어디인지
- 프로세스 마이닝 없이도 내가 아는 것
- 지금 안 움직이면 현장에서 벌어질 일
```

### Claude 답변 D

1. **내 눈으로 본 병목**: 봉제실 앞 대기가 체감 3일이다. 자재가 쌓여 있고, 봉제 조장이 "라인 비었어"라고 말하는 시점과 자재가 실제로 들어가는 시점 사이에 간극이 있다. 20년 봤으니까 안다.
2. **프로세스 마이닝 없이 아는 것**: 어맨드가 터지면 영업부가 생산부에 전달하는 데 하루, 생산부가 라인에 반영하는 데 하루 — 합치면 이틀이 그냥 날아간다. 이건 시스템 문제가 아니라 전화 한 통이면 줄일 수 있다.
3. **안 움직이면**: 다음 분기 Adidas 물량에서 납기 지연이 또 나온다. 작년에 3건이었는데 올해 5건 넘기면 바이어가 물량을 돌린다. 기다릴 시간이 없다.

### 프롬프트 — S (품질관리팀장 — 데이터와 규정의 눈)

```
나는 풍국산업의 품질관리팀장이야.
PuzzleData와 협업해서 프로세스 마이닝을 도입하려고 해.
너는 데이터와 규정의 관점에서 말하는 조언자야.

현장 베테랑은 "봉제실 앞 대기가 3일"이라고 하는데,
데이터로 보면 정말 그런가?
- ERP 로그에서 실제 대기 시간이 얼마인지
- 현장 경험과 데이터가 다를 수 있는 이유 3가지
- 검증 순서: 되돌릴 수 있는 것부터 먼저
```

### Claude 답변 S

**데이터로 보면:**
ERP 로그상 봉제실 앞 대기 시간은 평균 0.5일이다. 생산부장이 말하는 "3일"과 6배 차이가 난다.

**차이가 나는 이유:**
1. **수기 입력 지연** — 현장에서 작업 완료 후 몇 시간 뒤에 일괄 입력하는 관행. ERP 기록과 실제 공정 시각이 다르다.
2. **숨은 공정** — 부자재 attach, bonding 반복 등 ERP에 미등록된 공정이 존재. "대기"로 보이는 시간의 일부는 실제로는 비공식 작업 시간이다.
3. **용어 혼란** — "봉제 대기"를 현장은 "자재 도착~봉제 시작"으로, ERP는 "봉제 지시~봉제 시작"으로 잡는다. 측정 구간 자체가 다르다.

**검증 순서 (코모리의 원칙 — 되돌릴 수 있는 것부터):**
1. **1주차**: 용어 Lexicon 초안 20단어 — "대기"의 시작과 끝을 현장·ERP 양쪽에서 정의 (되돌릴 수 있음, 비용 = 0)
2. **2~3주차**: 스타일 1개의 P.O.에 대해, 현장 기록(수기)과 ERP 로그를 나란히 놓고 대조
3. **4주차 이후**: 대조 결과를 보고 프로세스 마이닝 전사 확대 여부 결정 (핵심 자산 = 전사 데이터, 가장 늦게)

### 차이 대조표 — 차이가 안개이고, 안개가 신호다

| 항목 | D(현장)가 본 것 | S(데이터)가 본 것 | 같은가 다른가? |
|:--|:--|:--|:--|
| **봉제실 대기** | 체감 3일 | ERP 0.5일 | **다름 = 안개 (6배 차이가 어디서 오는가?)** |
| **가장 큰 위험** | 납기 지연 → 바이어 이탈 | 데이터 품질 부족 → 잘못된 진단 | **다름 = 안개 (현장 위험 vs 분석 위험)** |
| **첫 번째 행동** | 지금 킥오프 | "대기"의 정의부터 맞추자 | **다름 — 그러나 양립 가능** |

> D만 따르면 보바리 조산 — 데이터 없이 뛴다.
> S만 따르면 살리나 마비 — 정의를 맞추다 1년이 간다.
> **차이가 가리키는 곳이 먼저 시험할 곳이다.**

### 행동 (내가 채우는 빈칸)

```
D와 S가 가장 다른 항목: 봉제실 대기 (체감 3일 vs ERP 0.5일)

시험할 것: 스타일 1개의 P.O.에 대해
   현장 수기 기록(자재 도착~봉제 시작)과 ERP 로그를 1주일간 나란히 기록
규모: 시간 2일 (생산부 조장 1명 + 총무 고봉석 + PuzzleData PM 김정환)
실패해도 내일 다시 시작할 수 있는가? Y
```

## 3장 💒 스테인드글라스새장 — Calibration Slider

> 인터랙티브 버전: Calibration Slider를 브라우저에서 열어 슬라이더를 직접 움직여보세요 (별도 제공).

> **병**: 확신이 증거를 차단함
> **약**: 확신을 교정하라

### 🔲 보편 템플릿

**Step 1 — 혼자 교정:**
```
나는 [내 회사/프로젝트]의 [역할]이야.
[추진 중인 일]의 성공을 위해 반드시 참이어야 하는 가정 3개를 뽑아줘.
→ 각 가정에 확신도(%)를 매긴다.
→ "각 가정이 틀릴 수 있는 이유 3가지를 알려줘."
→ 확신도를 다시 매긴다. 간극 = 빚.
```

**Step 2 — D/S 교차 검증:**
```
같은 가정을 두 관점에서 검증해줘.
D(현장 경험자): 이 가정이 현장에서 맞는가? 경험으로 답해줘.
S(데이터 분석자): 이 가정이 데이터로 맞는가? 숫자로 답해줘.
→ D와 S의 답이 다른 가정이 가장 큰 빚. 먼저 검증.
```

### 📋 풍국산업 예시

### 프롬프트 — 핵심 가정 추출

```
나는 경영학 수업에서 풍국산업 SCM 개선 프로젝트를 하는 대학생이야.
PuzzleData와 협업해서 프로세스 마이닝 파일럿을 제안하려고 해.

이 파일럿의 성공을 위해 반드시 참이어야 하는 가정 3개를 뽑아줘.
```

### Claude 답변

1. "오라클 ERP의 P.O.→생산 데이터가 실제 공정 순서를 반영한다" (타임스탬프가 정확하다)
2. "용어 Lexicon 20단어면 1개 스타일의 공정을 분류할 수 있다"
3. "파일럿 결과를 이상협 사장이 보고 전사 확대를 승인할 것이다"

### 프롬프트 — 반증 요청

```
각 가정에 대해, 틀릴 수 있는 이유 3가지를 알려줘.
```

### Claude 답변

**가정 1 — ERP 데이터 정확성:**
- 현장에서 수기 입력 → 실제 작업 후 몇 시간 뒤 일괄 입력하는 관행
- 어맨드 발생 시 ERP 수정이 3일 지연 → 데이터와 현실의 괴리
- 부자재 attach, bonding 반복 등 ERP에 미등록된 "숨은 공정" 존재 가능

**가정 2 — Lexicon 20단어 충분성:**
- 문성암 교수: "7대 공정 외에도 되게 많다." 20단어로 부족할 수 있음
- 영어·한국어·일본어 3중 매핑 필요 → 실질 60단어
- 바이어마다 같은 공정을 다른 이름으로 부름 (Adidas vs Nike 용어 차이)

**가정 3 — 사장 승인:**
- 이상협 사장은 "영업부 파워" 강조 — 생산 최적화보다 영업 매출 우선
- 파일럿 기간 중 대형 어맨드가 터지면 관심이 다른 곳으로 이동
- IT 도구 도입 후 방치한 전례 존재 가능

### 확신도 교정

| 가정 | 반증 전 | 반증 후 | 간극 = 빚 |
|:--|:--|:--|:--|
| ERP 데이터 정확성 | 80% | **45%** | 35%p — **가장 큰 빚. 먼저 검증** |
| Lexicon 20단어 충분 | 70% | **50%** | 20%p |
| 사장 승인 | 60% | **40%** | 20%p |

→ **첫 번째 행동**: 스타일 1개의 ERP 타임스탬프 vs 현장 실제 시각을 대조. 1일이면 된다.

### Step 2: D/S 교차 검증 — 같은 가정, 다른 눈

같은 가정을 D(현장)와 S(관리)가 각각 검증하면 답이 다르다.

| 가정 | D(현장)의 검증 | S(관리)의 검증 | 차이 = 빚의 소재지 |
|:--|:--|:--|:--|
| ERP 데이터 정확성 | "수기 입력을 몇 시간 뒤 일괄로 하니까 당연히 안 맞죠" | "타임스탬프 기준 정합성 93%인데요?" | **7%가 빠지는 곳이 첫 검증 대상** |
| Lexicon 20단어 충분 | "공정마다 사람마다 부르는 이름이 달라요. 20개로는 어림없어요" | "7대 공정 분류 기준으로는 20단어면 커버율 85%" | **현장 이름 vs 공식 분류의 괴리** |
| 사장 승인 | "사장님은 영업부 출신이라 생산 쪽 제안은 잘 안 봐요" | "전년 대비 납기 준수율 데이터를 붙이면 관심을 가질 것" | **설득 언어의 차이** |

→ D와 S가 가장 크게 갈리는 가정(ERP 정확성) = 가장 큰 빚 = 먼저 검증.
→ Step 1의 혼자 교정이 파늘루라면, Step 2의 교차 검증이 리외다.

## 4장 ⏱️ 시계새장 — 2×2 빈칸 찾기

> **병**: 성공했으니 바꿀 이유가 없음
> **약**: 다음 불을 피워라

### 🔲 보편 템플릿

**Step 1 — 빈칸 찾기:**
```
나는 [내 회사/프로젝트]의 [역할]이야.
[핵심 경쟁력 한 줄].
내 사업에서 가장 중요한 두 가지 차원을 뽑아줘.
2×2 매트릭스로 만들어서, 지금 내가 하는 것은 어느 칸인지,
비어있는 칸에 뭐가 들어갈 수 있는지 3가지를 말해줘.
```

**Step 2 — D/S로 빈칸 채우기:**
```
빈칸 [번호]에 대해 두 관점으로 답해줘.
D(현장 경험자): 우리가 이걸 할 수 있는가? 역량 기준으로.
S(데이터 분석자): 우리가 이걸 팔 수 있는가? 시장 기준으로.
→ "할 수 있다 vs 팔 수 있다"의 gap이 실험 설계의 출발점.
```

### 📋 풍국산업 예시

### 프롬프트

```
나는 경영학 수업에서 풍국산업을 분석하는 대학생이야.
풍국은 가방 OEM으로 30년간 성공해왔어.
바이어 커스터마이제이션(어맨드 30회 대응)이 핵심 경쟁력이야.

풍국의 사업에서 가장 중요한 두 가지 차원을 뽑아줘.
그리고 2×2 매트릭스로 만들어서,
지금 내가 하고 있는 것은 어느 칸인지,
비어있는 칸에 뭐가 들어갈 수 있는지 3가지 가능성을 말해줘.
```

### Claude 답변

**축:** X = 주문 유형(소량 커스텀 vs 대량 벌크), Y = 가치 제공(생산만 OEM vs 설계+생산 ODM)

```
              소량(커스텀)          대량(벌크)
          ┌─────────────────┬─────────────────┐
 ODM      │   ③ 빈칸        │   ② 빈칸        │
 (설계+   │ 소규모 브랜드에   │ 자체 브랜드      │
  생산)   │ 설계+생산 일괄    │ 대량 생산        │
          ├─────────────────┼─────────────────┤
 OEM      │ ★ 풍국 현재      │   ① 빈칸        │
 (생산만) │ (어맨드 대응)     │ 표준화 벌크      │
          └─────────────────┴─────────────────┘
```

**빈칸 가능성:**
1. **① 대량 OEM**: 표준화 벌크 생산. 마진 낮지만 볼륨 안정. 풍국이 안 하는 이유: "어맨드 대응"이 정체성이라 표준화를 경시.
2. **② 대량 ODM**: 자체 설계 + 대량 생산. 두산이 포목점에서 중공업으로 갔듯, 풍국이 "가방 설계 회사"로 재정의.
3. **③ 소량 ODM**: D2C 브랜드에 설계+생산 일괄 제공. 어맨드 대응 역량이 바로 이 칸의 경쟁력. 가장 가까운 빈칸.

### 프롬프트 — 담뱃대 크기의 실험

```
빈칸 ③ (소량 ODM)을 시험하려면,
1주일 안에 잃어도 괜찮은 수준으로 뭘 해볼 수 있어?
```

### Claude 답변

- PuzzleData PM 김정환에게 부탁: 지인 중 D2C 가방 브랜드 1곳에 "샘플 3개를 풍국이 설계+생산하면 관심 있는가?" 물어보기
- 비용: 시간 반나절, 돈 0원, 관계 1건
- 실패해도 내일 다시 시작할 수 있는가? **Y**

### Step 2: D/S로 빈칸 채우기 — 할 수 있다 vs 팔 수 있다

같은 빈칸을 D(현장)와 S(관리)가 채우면 답이 다르다.

| 빈칸 | D(현장)가 채운 것 | S(관리)가 채운 것 | 차이 = 실험의 출발점 |
|:--|:--|:--|:--|
| ① 대량 OEM | "우리 봉제 라인이 벌크도 소화 가능해요" | "벌크 마진율 5%로는 고정비를 못 덮어요" | **능력 vs 수익성** |
| ② 대량 ODM | "설계 인력이 없어요, 바이어 도면만 받아왔으니까" | "ODM 브랜드의 연 성장률 12%, 진입하면 마진 3배" | **역량 부족 vs 시장 기회** |
| ③ 소량 ODM | "재단팀이 이미 소량 커스텀을 잘하니까 바로 할 수 있어요" | "마진 2배인데 영업 채널이 없어요" | **할 수 있다 vs 팔 수 있다** |

→ ③에서 D는 "된다", S는 "팔 곳이 없다". 이 gap이 담뱃대 실험의 출발점.
→ 할 수 있다가 곧 해야 한다가 되고, 해야 한다가 곧 하고 있다가 되면 — 빈칸이 안 보인다. 그게 로빈슨의 시계다.
